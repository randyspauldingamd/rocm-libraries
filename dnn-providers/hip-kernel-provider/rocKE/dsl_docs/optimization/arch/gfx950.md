# Target Architecture Reference: gfx950 / CDNA4 / MI350X / MI355X

> Specializes the [CK DSL Optimization Runbook](../optimization_runbook.md).
> Sibling arches: see [arch/README.md](README.md).

The runbook itself is architecture-agnostic — every principle and
lever applies to any AMDGPU CDNA target. This file collects the
**gfx950 / CDNA4** specifics that the DSL targets by default. When
porting to gfx942 (MI300X), gfx940 (MI250X-class CDNA3), or gfx90a,
the corresponding caps and intrinsics differ — always verify against
the vendor spec sheet. (Section numbers below preserve the runbook's
historical `21.x` numbering so existing cross-references resolve.)

> **Backend-validity note.** Perf and register/occupancy numbers are
> backend-sensitive. The **production** backend is **llvm22 + comgr 7.2**
> (selected by importing `torch` first, so the bundled
> `torch/lib/libamd_comgr.so` resolves). Some pre-2026-06-19 measurements
> were taken on an **llvm20 + comgr 7.0.1** backend (when a dependency
> dlopened the system comgr before torch); on llvm20 the d128 attention
> register/occupancy and prefill-gap picture was materially worse. Such
> numbers are corrected inline below to the llvm22 values. Correctness /
> byte-identity / ISA-availability facts are backend-independent and
> unchanged.
>
> **Recommendation: always build + benchmark on `llvm22`** (import torch / a
> ROCm-7.2 toolchain *first* so comgr 7.2 loads; confirm `ROCKE_LLVM_FLAVOR`
> resolves to `llvm22`). The flavor is *not* perf-neutral — MFMA scheduling
> and VGPR/AGPR allocation differ, so `llvm20` can report **false**
> register-bound / AGPR-spill / 1-WG/CU regressions that vanish on `llvm22`.

### 21.1 MFMA Atom Catalog (gfx950)

The full DSL atom catalog lives in `helpers/atoms.py::MFMA_*_ATOMS`.
On gfx950 the following are available:

| Atom factory | A / B / C dtype | M × N × K | Notes |
|---|---|---|---|
| `MfmaAtom.f16_4x4x4` | fp16 / fp16 / fp32 | 4×4×4 | Many independent small problems per wave (used by `DirectConv4cSpec`) |
| `MfmaAtom.f16_16x16x16` | fp16 / fp16 / fp32 | 16×16×16 | Standard small tile |
| `MfmaAtom.f16_16x16x32` | fp16 / fp16 / fp32 | 16×16×32 | K-packed (recommended over `16x16x16`) |
| `MfmaAtom.f16_32x32x8` | fp16 / fp16 / fp32 | 32×32×8 | Larger output tile |
| `MfmaAtom.f16_32x32x16` | fp16 / fp16 / fp32 | 32×32×16 | K-packed + larger output tile. **C-output lane layout matches A-input layout** — chained MFMA without re-pack (§7.2) |
| `MfmaAtom.bf16_16x16x16` | bf16 / bf16 / fp32 | 16×16×16 | |
| `MfmaAtom.bf16_16x16x32` | bf16 / bf16 / fp32 | 16×16×32 | K-packed |
| `MfmaAtom.bf16_32x32x16` | bf16 / bf16 / fp32 | 32×32×16 | K-packed; chained-MFMA layout match |
| `MfmaAtom.fp8_*_f8` | fp8e4m3 / fp8e4m3 / fp32 | varies | CDNA4 native fp8 MFMA |
| `MfmaAtom.bf8_*_bf8` | bf8e5m2 / bf8e5m2 / fp32 | varies | CDNA4 native bf8 MFMA |
| Mixed-precision MFMA | mixed | varies | bf16 / fp8, fp16 / fp8, etc. |
| Scaled MFMA (block-scale) | fp8 + per-block scale | varies | `BlockScaleGemmSpec`; `helpers/atoms.py::MFMA_FP8_ATOMS` |
| MX (microscaling) MFMA | fp8 + E8M0 shared exponent | varies | `MxGemmSpec`; `helpers/mx_scale.py` |
| i8 / i4 atoms | int8 / int4 packed | varies | `helpers/codebook.py` for i4 unpack |

Lane-layout matching across chained atoms (§7.2): the 16×16 atoms'
C-output doesn't natively match their A-input, so a QK→PV-style
chain needs an LDS round-trip or a cross-lane permute. The 32×32
atoms natively match — prefer them when the chain pattern allows
(`use_mfma_32x32`, `use_transposed_qk_32x32` on the attention 2D path).

### 21.2 LDS specifics

| Spec | gfx950 | gfx942 (MI300X) | gfx90a (MI250X) |
|---|---|---|---|
| LDS per CU | 160 KB | 64 KB | 64 KB |
| LDS banks | 64 | 32 | 32 |
| Preferred swizzle (§6.4a) | padding | XOR | XOR |
| `ds_read_b64_tr_b16` / `ds_read_b128_tr_b16` | CDNA4 only | — | — |
| `ds_read_tr_b8` (i8 / fp8 / bf8 transposed) | CDNA4 only | — | — |
| `ds_read_b128` conflict period | 64 dwords | 32 dwords | 32 dwords |
| `ds_write_b128` conflict period | 32 dwords | 32 dwords | 32 dwords |
| Other `ds_read_b{32,64}` / `ds_write_*` conflict period | 32 dwords | 32 dwords | 32 dwords |
| Intra-dword sub-dword accesses | conflict-free | conflict-free | conflict-free |
| Unaligned `ds_read_u16` charged to | `SQ_LDS_IDX_ACTIVE` (not `SQ_LDS_BANK_CONFLICT`) | same | same |

Important asymmetry: `ds_write_b128` (32-dword period) and
`ds_read_b128` (64-dword period) on gfx950 can require **different
swizzle strategies** for reads vs writes. See `LDS_a.md` / `LDS_b.md`
in the empirical studies for the full opcode × phase table.

### 21.3 Cross-lane primitives

| Primitive | Available on | DSL surface | Notes |
|---|---|---|---|
| `v_permlane32_swap_b32` | gfx950 (CDNA4) | `permlane32.swap` intrinsic | Cheap VOP1 32-lane swap — the canonical primitive for FA-style softmax row reduction on CDNA4 |
| `ds_bpermute_b32` | gfx9* and gfx95* | `b.ds_bpermute` | LDS-bus cross-lane shuffle (costs `lgkmcnt`) |
| `ds_bpermute_b64` | synthesised from two b32 | `b.ds_bpermute_b64` | Cross-lane 64-bit shuffle |
| `ds_swizzle_b32` | gfx9* and gfx95* | `b.ds_swizzle_xor` | LDS-bus cross-lane shuffle |
| `ds_read_tr16_b64` | gfx950 | `b.ds_read_tr16_b64` | Transposed bf16/fp16 tile reader (4 rows per lane) |
| `ds_read_tr16_b128` | gfx950 | `b.ds_read_tr16_b128` | Transposed bf16/fp16 tile reader (8 rows per lane) |
| `ds_read_tr_b8` | gfx950 | `b.ds_read_tr_b8` | Transposed i8/fp8/bf8 tile reader |
| `s_setprio` | gfx9* and gfx95* | `b.s_setprio` | Wave priority hint |
| `s_sched_barrier`, `s_sched_group_barrier` | available in ISA | `b.sched_barrier`, `b.sched_group_barrier` | **Compile-time scheduling fence — distinct from a runtime instruction.** `b.sched_barrier(mask)` lowers to `llvm.amdgcn.sched.barrier`, which constrains the LLVM post-RA scheduler at compile time. The fence does its work during codegen and frequently leaves **no `s_sched_barrier` instruction in the disassembly** — so a zero `sched_barrier` sub-bucket in `probe_isa_inspect.py` means *the runtime instruction is absent*, **not** that the directive had no effect (see §21.8). |
| `s_barrier` (bare, no implicit waitcnt) | gfx9* and gfx95* | `b.s_barrier_bare` | Pure WG rendezvous; caller issues the explicit `s_waitcnt`. Use for single-barrier ping-pong loops (§8.6). Named/split barriers (`s.barrier.signal/wait`) **ICE the gfx950 backend** — do not emit. |
| `iglp_opt` | gfx9* and gfx95* | `b.iglp_opt(n)` | Canned backend GEMM MFMA/memory interleave (`llvm.amdgcn.iglp.opt`). *Survives* on gfx950 (unlike `sched_barrier`). Helps schedule-bound loops with `ds_write` traffic; neutral on direct-to-LDS / barrier-bound loops (§8.4, §8.6). |

`probe_occupancy.py` uses these caps when reporting waves/CU and

`MAX_WAVES_PER_CU`).

#### 21.4a Worked example: the d128 attention-prefill body is occupancy-clean on llvm22 — the residual is MFMA scheduling

The single-batch d128 attention prefill combo (`UnifiedAttention2DTiledSpec`,
`head_size=128`, `num_seqs=1`, no-FP8, no-SW, `max_seqlen_q>256`) is the
canonical case where **reading the limiter correctly chooses the lever**.
On the production **llvm22** backend the shipped `BLOCK_M=128`
`use_q_direct_reg` body is already register- and LDS-clean — it sits at
**2 WG/CU with room to spare**:

| Config (d128, production llvm22) | VGPR | AGPR | spill | LDS | WG/CU |
|---|---:|---:|---:|---:|---:|
| `BLOCK_M=128`, `use_q_direct_reg` (shipped body) | **213** | 0 | 0 | 32 KB | **2** |
| + `use_softmax_mfma_interleave` (the winning config) | 240 | 0 | 0 | 32 KB | **2** |

Both numbers were extracted with `llvm-readelf` on the produced HSACO
(throttle-independent — the primary occupancy signal on a clock-throttled
box; see runbook §10.6). `waves_per_eu=2` does the occupancy work; the
register file is well under the 256 cap at 2 waves/SIMD.

> **Backend caveat.** An earlier llvm20 measurement of this body reported
> **256 VGPR with spills and only 1 WG/CU**, and motivated an LDS-cut
> occupancy story (a `T=block_size` "small tile" vs a `T=64`
> K-single-buffer lever, with a 229 / 173 / 215 VGPR table). That
> occupancy wall was **an llvm20 backend artifact**: on production llvm22
> the body is already 213 VGPR / 0 spill / 2 WG/CU, so there is no wall to
> cut. The `HIPDNN_GFX950_D128_SMALL_TILE` escape hatch and the
> single-buffer/small-tile levers remain in the tree but are **not** the
> production lever.

With occupancy clean, the real residual is **MFMA scheduling**: the
mainloop's max inter-MFMA instruction gap (the MFMA-idle stretch where the
causal-mask + softmax VALU runs) is the cost to hide. The production lever
is `use_softmax_mfma_interleave` (one `iglp_opt` at the loop top — §21.3,
runbook §8.4), which lets the backend interleave that VALU mass into the
MFMA window and shrinks the gap; it is what crosses Triton at S1024 (see
the [attention 2D experiment summary](../../architecture/attention_2d_experiment_summary.md)).
The `use_agpr_alloc_zero` / mfma-vgpr-form lever is bit-accurate but **not**
the lever here on llvm22 (it leaves the gap unchanged). The remaining gap
is the causal-mask VALU mass that even the canned interleave cannot fully
overlap, plus the HBM-bandwidth-bound long-context regime at S≥2048.

### 21.8 Compiler caveats specific to gfx950

- **`s_sched_barrier` is a compile-time scheduling fence, not a
  reliably-emitted runtime instruction — do not read "0 in the ISA
  bucket" as "no effect".** Earlier notes said the backend "silently
  removes" `s_sched_barrier` / `s_sched_group_barrier` on gfx950; that
  was a misread of the disassembly. `b.sched_barrier(mask)` lowers to
  `llvm.amdgcn.sched.barrier`, which steers the LLVM post-RA scheduler
  *at compile time* and often leaves no corresponding instruction in the
  object — `probe_isa_inspect.py` reporting a `sched_barrier` sub-bucket
  of 0 confirms the runtime instruction is absent, not that the
  scheduling constraint was dropped. The fence has a measurable effect:
  placing it between the QK MFMA cluster and the post-QK async prefetch
  steers the scheduler to keep the QK MFMAs packed instead of interleaving
  the next-tile `buffer_load_lds`. An early **llvm20** measurement of the
  single-batch d128 prefill combo at `num_warps==1` recorded a same-session
  **+35–36 %** from this fence (and a −2.5–4.6 % regression at
  `num_warps>=2`, where the extra ILP makes the fence over-constrain the
  scheduler). That win is **both backend-specific and superseded**: on the
  production **llvm22** backend the d128 body is already occupancy-clean at
  `num_warps=2` (§21.4a), and the production schedule lever is
  `use_softmax_mfma_interleave` (an `iglp_opt`), which is mutually exclusive
  with `use_sched_barrier`. So the `sched_barrier` lever is **not on any
  live attention cohort** — but the *mechanism* (a compile-time fence with a
  real effect, distinct from a runtime instruction) is the accurate and
  backend-independent description. The separate
  `sched_group_barrier` directives the compv4/compv3 GEMM pipeline emits
  are net-negative on gfx950 and are off by default (runbook §8.4).
  Always confirm a hint's effect by diffing the mainloop ISA, never by
  perf alone.
- **`-mllvm -enable-post-misched=0`** is safe in CK's CMake-generated
  code but has been observed miscompiling MLIR-generated kernels on
  this arch — treat as risky.
- The remaining recommended CK-style flag stack
  (`-mllvm -amdgpu-function-calls=false`,
  `-mllvm -amdgpu-early-inline-all=true`,
  `-mllvm --lsr-drop-solution=1`, `-fno-offload-uniform-block`,
  `-O3`, fast / unsafe math) has been measured safe but rarely
  moves the needle by more than 1 % on gfx950 — large gaps come
  from kernel structure, not flags (§10.2).

### 21.9 Default ISA target

| Spec | Value |
|---|---|
| `compile_kernel(...)` default ISA | `amdgcn-amd-amdhsa--gfx950` |
| `_DATALAYOUT` in `core/lower_llvm.py` | gfx950 datalayout string baked in |
| Other supported targets | `gfx942` (MI300X), `gfx90a` (MI250X) — verify MFMA atom shapes are valid for the target |

To target a different arch from the default, pass
`isa="amdgcn-amd-amdhsa--gfx942"` (or similar) to `compile_kernel`,
and verify the atoms picked by the spec are in that arch's
`_F16_WARP_TILE_SHAPES_*` / `_BF16_WARP_TILE_SHAPES_*` set.
