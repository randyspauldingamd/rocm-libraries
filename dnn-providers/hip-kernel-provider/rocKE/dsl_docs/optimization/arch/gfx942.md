# Target Architecture Reference: gfx942 / CDNA3 / MI300X / MI325X

> Specializes the [CK DSL Optimization Runbook](../optimization_runbook.md).
> Sibling arches: see [arch/README.md](README.md).

The runbook itself is architecture-agnostic — every principle and lever
applies to any AMDGPU CDNA target. This file collects the **gfx942 /
CDNA3** specifics (MI300X / MI325X-class). The facts below are grounded
in AMD's public docs — the [AMD Instinct MI300 / CDNA3 ISA Reference
Guide](https://www.amd.com/content/dam/amd/en/documents/instinct-tech-docs/instruction-set-architectures/amd-instinct-mi300-cdna3-instruction-set-architecture.pdf)
(cited as "CDNA3 ISA §x"), the [ROCm GPU hardware specs](https://rocm.docs.amd.com/en/latest/reference/gpu-arch-specs.html),
and the [AMD matrix-instruction calculator](https://github.com/ROCm/amd_matrix_instruction_calculator).
Measured / kernel-specific findings live in the case study (§21.10).
(Section numbers below preserve the runbook's historical `21.x`
numbering so existing cross-references resolve.)

> **Backend-validity note.** Perf and register/occupancy numbers are
> backend-sensitive. The **production** backend is **llvm22 + comgr 7.2**
> (selected by importing `torch` first). The d128 attention
> register/occupancy and prefill-gap numbers in §21.4a and the case study
> were measured on an **llvm20 + comgr 7.0.1** backend (when a dependency
> dlopened the system comgr before torch). On gfx950 that same llvm20→llvm22
> switch erased an apparent register/occupancy wall, so the gfx942 d128
> AGPR-residency / occupancy findings below are flagged as **backend-dependent
> and pending production-llvm22 re-validation** rather than restated as final.
> ISA-availability and correctness facts are backend-independent.
>
> **Recommendation: always build + benchmark on `llvm22`** (import torch / a
> ROCm-7.2 toolchain *first* — even on a system-ROCm-7.1 gfx942 node, the
> torch-bundled comgr 7.2 overrides it; confirm `ROCKE_LLVM_FLAVOR`→`llvm22`).
> The flavor is *not* perf-neutral: `llvm20` reported the AGPR-spill / 1-WG/CU
> picture below, which `llvm22`'s VGPR-form MFMA may erase entirely.

### 21.1 MFMA Atom Catalog (gfx942)

The DSL atom catalog lives in `helpers/atoms.py`; which atoms are legal
on an arch is data-driven from `core/arch/data/arch_specs.json` and
enforced by `instances/common/gemm_universal.py::is_valid_spec` /
`op_for_shape`. The gfx942 catalog the DSL exposes:

| Atom factory | A / B / C dtype | M × N × K | Notes |
|---|---|---|---|
| `MfmaAtom.f16_16x16x16` | fp16 / fp16 / fp32 | 16×16×16 | Standard tile |
| `MfmaAtom.f16_32x32x8` | fp16 / fp16 / fp32 | 32×32×8 | Widest dense **f16** atom on CDNA3 |
| `MfmaAtom.bf16_16x16x16` | bf16 / bf16 / fp32 | 16×16×16 | Standard bf16 tile |
| `MfmaAtom.fp8_*_f8` | fp8e4m3 / fp8e4m3 / fp32 | 16×16×32, 32×32×16 | CDNA3 native fp8 MFMA (wide-K) |
| `MfmaAtom.bf8_*_bf8` | bf8e5m2 / bf8e5m2 / fp32 | 16×16×32, 32×32×16 | CDNA3 native bf8 MFMA (wide-K) |

**gfx950-only (rejected on gfx942 by `is_valid_spec`):** the wide-K
**f16 / bf16** atoms `16x16x32` and `32x32x16`. Wide-K on CDNA3 exists
for **fp8 / bf8 only** — there is no dense wide-K f16/bf16 path.

**Not present on CDNA3 at all:** block-scale / microscaling (MX) MFMA and
FP6 / FP4 — these are CDNA4 (gfx950) additions
([ROCm Matrix-Cores blog](https://rocm.blogs.amd.com/software-tools-optimization/matrix-cores-cdna/README.html)).

The full CDNA3 *hardware* MFMA set — including the small `4x4x4` /
`16x16x4` atoms (used by `DirectConv4cSpec`), i8 MFMA, and the sparse
SMFMAC ops — is enumerated by the matrix calculator (`--architecture
cdna3 --list-instructions`); the DSL exposes the subset above.

### 21.2 LDS specifics

| Spec | Value | Source |
|---|---|---|
| LDS per CU | **64 KB** | CDNA3 ISA §2.2.1 / §11.1 |
| LDS banks | **32** (512 entries × 4 B each) | CDNA3 ISA §2.2.1 / §11.1 |
| Transpose-read (`ds_read_*_tr_*`) | **none** (CDNA4-only) — see §21.3 | `arch_specs.json: has_ds_read_tr=false` |

The XOR-vs-padding swizzle *selection method* is arch-neutral (runbook
§6.4a); plug in the 64 KB / 32-bank capacity above. A consequence of the
32-bank geometry: 64 words across 32 banks = a 2-word/bank floor, so a
fully conflict-free transposed LDS layout is impossible without a
transpose-read (which this arch lacks). AMD documents LDS bank conflicts
only qualitatively (CDNA3 ISA §11.1); measured per-opcode conflict
periods are in the case study (§21.10).

### 21.3 Cross-lane primitives

Availability is from the CDNA3 ISA; the DSL-surface column lists the IR
builder where one exists.

| Primitive | On gfx942? | DSL surface | Notes |
|---|---|---|---|
| `v_perm_b32` (in-register byte permute) | yes (ISA op) | `b.perm_b32` | In-register 2×2 f16 / 4×4 i8 transpose at LDS-store time (CK `transpose_vectors`); pure VALU, no LDS port, no `lgkmcnt` |
| `ds_bpermute_b32` | yes | `b.ds_bpermute` | LDS-bus cross-lane shuffle — **costs `lgkmcnt`** |
| `ds_swizzle_b32` | yes | `b.ds_swizzle_xor` | LDS-bus cross-lane shuffle — **costs `lgkmcnt`** |
| `s_setprio` | yes | `b.s_setprio` | Wave priority hint |
| `s_barrier` | yes | `b.s_barrier_bare` | WG rendezvous (caller issues `s_waitcnt`) |
| `iglp_opt` | yes | `b.iglp_opt(n)` | Canned backend GEMM MFMA/memory interleave |
| `v_permlane16/32_swap_b32` | **NO — gfx950-only** | — | CDNA4 feature ([LLVM PR #117260](https://github.com/llvm/llvm-project/pull/117260)) |
| `ds_read_*_tr_*` (transpose reads) | **NO — gfx950-only** | — | Absent from the CDNA3 ISA |

For in-register reshapes prefer the VALU byte-permute (`b.perm_b32`) over
`ds_swizzle` / `ds_bpermute` — the latter are LDS-port ops that cost
`lgkmcnt` and serialize against LDS traffic (measured impact: case study,
§21.10). Reach for them only for genuine cross-lane traffic with no VALU
equivalent.

### 21.4 Register / occupancy

| Quantity | Value | Source |
|---|---|---|
| SIMDs / CU | 4 | CDNA3 ISA / ROCm specs |
| Waves / SIMD (hardware max) | **10** | HIP hardware-implementation ("up to ten warps"; 8 was CDNA2) |
| Waves / CU (hardware max) | **40** | = 10 × 4 |
| VGPR file / SIMD | **512 total**, split ≤256 architected + ≤256 AGPR | CDNA3 ISA §3.6.4 |
| SGPRs / wave | 16–102 (allocated in units of 16) | CDNA3 ISA §3.6.2 |
| VGPR allocation granularity | 8 dwords (AMD tuning guides cite 16) | CDNA3 ISA §3.6.4 |
| Threads / CTA (max) | 1024 | CDNA3 ISA §4.3 |
| Wave size | 64 (wave64) — the only path the DSL helpers support | CDNA3 ISA |

`probe_occupancy.py --arch gfx942` reports the apparent limiter (`VGPR`,
`AGPR`, `LDS`, `WAVES_PER_EU_HINT`, `MAX_WAVES_PER_CU`). Note it
currently models a conservative **8** waves/SIMD and 16-VGPR granularity
(CDNA2-era values) — AMD documents 10 waves/SIMD for CDNA3. The
kernel-specific LDS-vs-occupancy budget analysis is in the case study
(§21.10).

#### 21.4a Worked example: d128 attention prefill occupancy / scheduling (llvm20-measured — re-validation pending)

> **Backend caveat (read first).** The register/occupancy and prefill-gap
> numbers in this subsection were measured on the **llvm20 + comgr 7.0.1**
> backend, not the production **llvm22 + comgr 7.2**. On gfx950 the
> equivalent d128 body went from "256 VGPR + spills, 1 WG/CU" on llvm20 to
> "213 VGPR, 0 spill, 2 WG/CU" on llvm22 — i.e. the apparent register/AGPR
> wall was a backend artifact. The gfx942 AGPR-residency → 1-WG/CU finding
> is therefore **backend-dependent and pending production-llvm22
> re-validation** (mfma-vgpr-form may eliminate the spill here too, as it
> did on gfx950). Treat the absolute `0.53–0.90×`-vs-flash gap below as an
> llvm20 reading, not a settled production result.

On the **llvm20** backend the shipped gfx942 d128 attention prefill wide
path (transposed-x8 with K **single-buffered**) read as **2 WG/CU** —
`llvm-readelf` on the produced HSACO (throttle-free):

| Config (d128 wide, llvm20) | LDS | V+A regs | WG/CU |
|---|---:|---:|---:|
| transposed-x8, K single-buffered (shipped) | 32 KB (`K_lds[1]` 16 KB + `V_lds[1]` 16 KB) | 340 | **2** |

The old "d128 wide = 48 KB → 1 WG/CU, needs a narrower-N accumulator
redesign" framing was wrong even on llvm20: K single-buffering already
halved `K_lds`. A large fraction of the 340 V+A figure was **accumulator
residency in the AGPR file**, which is exactly the quantity the llvm22
backend reorganizes — so this count must be re-measured on production before
it is cited as the occupancy state. A small-tile `T=32` **double-K** variant
(LDS 24 KB) gave a robust **+11–14 %** at S≤2048 (both dtypes) on llvm20 —
via a *different* mechanism than occupancy: it restores the K
**double-buffer** prefetch that the shipped `T=64` path gives up to fit LDS.
That is a prefetch/scheduling win. **Caveat:** `T=32` requires a paged-KV
cache `block_size=32` (`tile_size` must be a multiple of `block_size`); if
production cache is `block_size=64`, `T=32` is illegal and the win must be
re-validated.

On llvm20 the measured standing was that **gfx942 d128 prefill still loses
to flash** (best ~0.53–0.67× bf16, ~0.67–0.90× fp16). Given the gfx950
llvm20→llvm22 reversal, **these ratios must be re-measured on production
llvm22 before being treated as the gfx942 gap** — the residual was
attributed to inner-loop scheduling (the CK Tile techniques: conflict-free V
via a store-path transpose, direct-to-LDS async, staggered partial
`lgkmcnt` waits), which remains the right lever family regardless of
backend. gfx942's wide bf16/fp16 atom is `mfma_f32_32x32x8` (K=8); the K=16
atom and `ds_read_tr16` that gfx950 uses for conflict-free V are **not
available** here. Full per-experiment detail (incl. the wide4 WG=256 ship
and the parked conflict-free-V store lever) is in the
[gfx942 attention case study](../../architecture/attention_2d_gfx942_experiment_summary.md);
the occupancy methodology on a throttled box is in runbook §10.6.

### 21.6 Buffer descriptor (AMDGPU)

| Spec | Value | Notes |
|---|---|---|
| Buffer resource descriptor (V#) DW3 fields | TYPE / DATA_FORMAT / NUM_FORMAT | CDNA3 ISA §9.1.8, Table 47 |

### 21.7 FP8 / quantization (gfx942)

CDNA3 native fp8 uses the **FNUZ** variants (not the OCP variants used by
CDNA4 / gfx950).

| dtype (CDNA3) | Range | Inf / −0 | Exp bias | DSL surface |
|---|---|---|---|---|
| **E4M3FNUZ** | ±240 | none | 8 | `Type("fp8e4m3")` / `QDType="fp8e4m3"` |
| **E5M2FNUZ** | ±57344 | none | 16 | `Type("bf8e5m2")` / `QDType="bf8e5m2"` |
| I8 saturating | ±127 | — | — | `QDType="i8"` |

The DSL surface names `fp8e4m3` / `bf8e5m2` resolve to the **fnuz**
hardware formats on gfx942. **Porting caveat:** CDNA4 / gfx950 uses the
OCP formats (E4M3FN, bias 7, larger ±448 max; E5M2 *with* Inf), so the
same byte read under the wrong dialect is off by a factor of two —
re-quantize when moving fp8 data between gfx942 and gfx950
([HIP fp8 docs](https://rocm.docs.amd.com/projects/HIP/en/latest/reference/fp8_numbers.html),
[ROCm Matrix-Cores blog](https://rocm.blogs.amd.com/software-tools-optimization/matrix-cores-cdna/README.html)).
There is **no block-scale / MX path** on gfx942 (§21.1).

### 21.8 Compiler / ISA constraints specific to gfx942

- **No wide async-LDS-DMA.** Load-to-LDS is 1 dword on CDNA3 (§21.6,
  CDNA3 ISA §9.1.9); wide b96 / b128 load-to-LDS is a CDNA4 addition.
- **No transpose-read** and **no `permlane*_swap`** (§21.3) — a
  transposed LDS operand must be built on the store side (`b.perm_b32` +
  store), not the read side.

The general IR-size ↔ comgr-walltime principle and the loop-roll fix live
in [§10.3](../optimization_runbook.md) and
`runtime/comgr_and_hipmodule.md`. Measured gfx942 compile-time budgets
and native-vector guidance are in the
[case study](../../architecture/attention_2d_gfx942_experiment_summary.md).

| Spec | Value |
|---|---|

When targeting gfx942 from the gfx950 default, the spec validator
(`is_valid_spec` / `op_for_shape` over `arch_specs.json`) rejects atoms
not in the gfx942 catalog (no dense wide-K f16/bf16, no MX) — see §21.1.
