# Pattern Type System

## Overview

StinkyTofu's pattern system supports multiple types of patterns, each designed for different kinds of code transformations and optimizations. Each pattern is explicitly labeled with its type to enable proper categorization, code generation, and future extensibility.

## Pattern Type Syntax

All patterns must be declared with an explicit type keyword:

```
<pattern_type> pattern PatternName {
  match { ... }
  constraints { ... }
  rewrite { ... }
}
```

### Example

```
peephole pattern AddFMAFusion_F32 {
  match {
    $fma_inst = v_fma_f32 $fma_result, $a, $b, $fma_const
    $add_inst = v_add_f32 $add_dst, $add_const, $fma_result
  }

  constraints {
    HasOneUse($fma_result)
    IsConstant($fma_const)
    IsConstant($add_const)
  }

  rewrite {
    $new_const = AddConstants($fma_const, $add_const)
    $new_fma = v_fma_f32 $add_dst, $a, $b, $new_const
    replace $fma_inst with $new_fma
    remove $add_inst
  }
}
```

## Pattern Types

### 1. Peephole Patterns

**Keyword**: `peephole`

**Purpose**: Local instruction-level optimizations that match and rewrite small sequences of instructions.

**Characteristics**:
- Bottom-up matching (root instruction first)
- Typically 2-3 instruction sequences
- Fast local transformations
- Applied within basic blocks
- No global dataflow analysis required

**Use Cases**:
- Constant folding (e.g., `v_add` + `v_fma` -> `v_fma` with folded constant)
- Instruction fusion (e.g., `v_mul` + `v_mul` -> `v_mul` with constant product)
- Algebraic simplifications
- Dead code elimination patterns

**Implementation**:
- Generated by TableGen from `.pattern` files
- Applied by `PeepholeOptimizationPass`
- Uses def-use analysis for constraints

**Example**:
```
peephole pattern MULMULFusion_F32 {
  match {
    $mul_inst1 = v_mul_f32 $result1, $const1, $a
    $mul_inst2 = v_mul_f32 $result2, $const2, $result1
  }

  constraints {
    HasOneUse($result1)
    IsConstant($const1)
    IsConstant($const2)
  }

  rewrite {
    $new_const = MulConstants($const1, $const2)
    $new_mul = v_mul_f32 $result2, $new_const, $a
    replace $mul_inst1 with $new_mul
    remove $mul_inst2
  }
}
```

## Implementation Details

### Pattern Type Enum

The pattern type is represented by the `PatternType` enum:

```cpp
enum class PatternType {
    Peephole   // Peephole optimization pattern
};
```

### Pattern Structure

Each parsed pattern includes its type:

```cpp
struct Pattern {
    std::string name;
    PatternType type;                        // Type of pattern (must be explicitly set)
    std::vector<MatchStmt> match;
    std::vector<Constraint> constraints;
    std::vector<RewriteStmt> rewrite;
};
```

### Lexer and Parser

The lexer recognizes pattern type keywords:

```cpp
// In IRLexer.hpp
enum class TokenKind {
    // ...
    KW_peephole,     // 'peephole' (pattern type)
    KW_pattern,      // 'pattern'
    // ...
};
```

The parser handles pattern type prefixes:

```cpp
Pattern PatternParser::parsePattern() {
    Pattern pattern;

    // Parse required pattern type
    if (lexer.peek().kind == TokenKind::KW_peephole) {
        pattern.type = PatternType::Peephole;
        lexer.consume();
    } else {
        error("Expected pattern type keyword (e.g., 'peephole')");
        return pattern;
    }
    // Future: Add support for other pattern types

    // Expect 'pattern' keyword
    if (lexer.peek().kind != TokenKind::KW_pattern) {
        error("Expected 'pattern' keyword after pattern type");
        return pattern;
    }
    // ...
}
```

## Adding New Pattern Types

This section shows how the `peephole` pattern type was added as a concrete example. Follow these same steps to add new pattern types in the future.

### Example: Adding the Peephole Pattern Type

1. **Add keyword to lexer** (`src/serialization/asm/IRLexer.hpp` and `.cpp`):
   ```cpp
   // In IRLexer.hpp
   KW_peephole,     // 'peephole' (pattern type)
   ```
   ```cpp
   // In IRLexer.cpp
   if(text == "peephole")
       return TokenKind::KW_peephole;
   ```

2. **Add enum value** (`include/stinkytofu/serialization/asm/PatternParser.hpp`):
   ```cpp
   enum class PatternType {
       Peephole   // Peephole optimization pattern
   };
   ```

3. **Update parser** (`src/serialization/asm/PatternParser.cpp` - implementation):
   ```cpp
   if (lexer.peek().kind == TokenKind::KW_peephole) {
       pattern.type = PatternType::Peephole;
       lexer.consume();
   }
   ```

4. **Update pattern files** (`lib/Dialect/Transforms/PeepholePatterns.pattern`):
   ```
   peephole pattern AddFMAFusion_F32 {
     match { ... }
     constraints { ... }
     rewrite { ... }
   }
   ```

5. **Pattern matcher generation** is handled automatically by existing TableGen code in `tools/tablegen/GenPatterns.cpp`.

6. **Pattern application** uses the existing `PeepholeOptimizationPass` in `src/transforms/asm/PeepholeOptimizationPass.cpp`.
